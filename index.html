Simple tools to check if your computer's CPU or Memory is getting overwhelmed.

If your server hard drive hits 100% full, use this combined command to instantly find the top 5 biggest folders eating your space:

Watching log files live to catch runtime errors or system crashes as they happen.

  • head /var/log/system.log — Read just the first few lines of a log file to see its setup.

Quick reachability tests to see if your server can talk to the internet or outside network hubs.

⚙️ Process Management & App Signals

App Process Types

  • Foreground Apps — Programs running directly in your active terminal. They block you from typing anything else until they finish.
  • Background Apps — Tasks running quietly behind the scenes so you can keep typing commands (you trigger this using an & symbol).
  • Daemons — Background workers that start automatically when the computer turns on (like web servers).

Process Status States

  • R Running — The app is actively thinking or ready to work.
  • S Sleeping — The app is idling, waiting for you to click or type.
  • D Deep Sleep — The app is stuck waiting for hardware or a disk save.
  • Z Zombie — A dead app that finished running but is stuck in memory.

Close & Kill Signals

  • SIGINT (Signal 2) — The manual cancel command you trigger from your keyboard when you press Ctrl+C.
  • SIGTERM (Signal 15) — A polite close request. Tells an app to save its work, stop traffic, and close down clean.
  • SIGKILL (Signal 9) — An instant force-kill command. It pulls the plug on frozen apps instantly.

💾 What is Swap Space?

When your high-speed RAM runs completely out of room, the computer uses a slow piece of your hard drive as a temporary backup memory space.

Swap keeps your system alive during traffic spikes instead of letting applications crash with an "Out of Memory" error.

  • sysctl vm.swapusage — Command to check how much backup hard-drive memory your Mac is using.

🌐 Networking & Subnetting Core Master Notes

🔢 IP Fundamentals & Private Boundaries

An IPv4 address looks like four numbers to us, but to a computer, it is a string of 32 binary bits (ones and zeros). Every IP address is divided into four sections called octets, and each section is exactly 8 bits long ($8 \times 4 = 32$ total bits).

🧠 The Math Behind 255: In binary, each position in an 8-bit slot has a specific placement value, doubling each time from right to left: 128, 64, 32, 16, 8, 4, 2, 1. When all 8 bits are turned on, they total 255.

🔒 Private Address Ranges (RFC 1918) — Memorize These!

These ranges are reserved strictly for internal private networks and are never routed across the public internet:

10.0.0.0/8 Range: 10.0.0.0 – 10.255.255.255 (16.7M IPs) AWS VPCs / Large Enterprises
172.16.0.0/12 Range: 172.16.0.0 – 172.31.255.255 (1M IPs) Docker Bridge Network Default
192.168.0.0/16 Range: 192.168.0.0 – 192.168.255.255 (65K IPs) Your Home/Office Router Default

📊 CIDR Notation Quick Reference

The slash notation (e.g., /24) defines the "subnet length"—telling the computer exactly how many bits belong to the network infrastructure and cannot be changed.

CIDR Prefix Total Usable IPs Practical DevOps Context Example
/8 16,777,214 10.0.0.0/8 — Massive entire corporate network foundation / global core routing block
/16 65,534 10.0.0.0/16 — Typical AWS VPC or Azure Virtual Network (VNet) root configuration
/24 254 10.0.1.0/24 — Standard everyday subnet for structural app or database tier groupings
/28 14 10.0.1.0/28 — Small isolation network segment (e.g., dedicated secure administrative jumps)
/32 1 10.0.1.45/32 — A single specific host / precise firewall Security Group inbound whitelist rule

💡 Subnetting Decoded (The Stencil Method)

An IP Address and a Subnet Mask are not separate entities. They are a permanent matching pair. Think of the IP address like a phone number, and the Subnet Mask like a rule defining the boundaries of the Area Code (neighborhood) versus the Local Line Number (the exact device).

🎯 Wherever the mask says 255: That position is completely locked down. It tells you the shared neighborhood root configuration.
🎯 Wherever the mask says 0: That field is free. It represents the space available to assign custom machine numbers.

📋 The Real-World Mapping Blueprint

If your workspace environment machine features an IP of 192.168.1.50 sitting alongside a Subnet Mask of 255.255.255.0, the network system layers them together like an overlay stencil:

  • The first three blocks match with 255.255.255, locking 192.168.1. securely as the immutable neighborhood route.
  • The final number aligns over the 0, identifying the machine's individual persona sequence as .50.
  • Because the final coordinate group is wide open, this specific room contains space to track running nodes numbering anywhere from .1 up to .254.

🏢 DevOps Architecture: Building Digital Security Walls

We leverage isolated subnet patterns in cloud deployment to isolate critical layers from unauthorized movement:

🌴 Subnet 1: Public Lobby

Range: 10.0.1.X

Open to the public internet. Houses entry load balancers and customer-facing Nginx web nodes to capture user traffic securely.

⚙️ Subnet 2: Private Workshop

Range: 10.0.2.X

Hidden from the external world. Holds core running application backend runtime APIs. Explicitly limited to speak strictly with Subnet 1.

🔒 Subnet 3: Secure Vault

Range: 10.0.3.X

Maximum lockdown environment. Holds sensitive customer database clusters. Rejects all connection rules unless originating from Subnet 2.

🛡️ The Security Win: If a vulnerability breaks a public internet proxy node in the Lobby, an attacker remains trapped inside that isolated subnet room. They have no route to navigate into the Vault because firewalls drop cross-subnet requests automatically.

⚠️ Azure Cloud Subnet Trap: The 5 Reserved IPs

While standard physical networks consume 2 address positions for bookkeeping, Azure explicitly stakes ownership over 5 IP addresses out of every subnet block. In a standard 256-IP capacity /24 subnet, you retain exactly 251 usable system slots.

🐋 Docker Containerization & Sub-System Modules

📦 The Core Definitions

Crucial baseline layers to map your infrastructure concepts:

  • Dockerfile: A plain text script containing step-by-step assembly blueprints (the recipe).
  • Docker Image: An immutable, read-only cached blueprint snapshot containing application layers and system runtimes.
  • Docker Container: A live, operational, isolated workspace execution engine built from an image template.

🐧 Why Alpine Linux?

The golden standard for production grade micro-containers:

  • Ultra-Lightweight Size: Total footprint is only ~5MB (vs. Ubuntu's ~78MB image floor), causing lightning-fast pull times.
  • Minimized Attack Surface: Stripped of unneeded background utilities, optimizing cloud platform application security.
  • Package Manager: Employs apk (e.g., apk add curl) instead of heavy apt layouts.

🐍 Why Python in DevOps?

The operational king of system infrastructure scripts:

  • High Readability: Reads closely to plain English, keeping developer cognitive load minimal.
  • Cloud Glue Frameworks: Native packages like AWS boto3 allow effortless resource orchestrations.
  • Script Boundary Rule: Use Bash for simple, sequential line loops; switch to Python for logic rules or JSON/API parsing blocks.

🐚 Zsh Terminal Bug Alert: Exclamation Quirk Fixes

The Issue: Running commands like echo "hello!" in Mac's default Zsh terminal throws zsh: event not found. This occurs because the ! character triggers history expansion inside double quotes.

Three Ways to Overcome This:

  • Use single quotes: echo 'hello!'
  • Escape with a backslash: echo "hello\!"
  • Disable history expansion features globally: setopt no_bang_hist

⚠️ DevOps Pitfall Alert: The Image Removal Sequence Trap

Docker refuses to delete an image via docker rmi if any active or stopped container structure depends on it. To execute a complete workspace sweep, you must complete the steps in this specific order: 
docker stop my-container    # Step 1: Safely shut down the active execution engine process thread
docker rm my-container      # Step 2: Delete the container record container box shell footprint layout
docker rmi rahulambadi/rahulrepo:v1 # Step 3: Now the underlying image layer can be purged from local storage safely.

🐙 Multi-Container Orchestration with Docker Compose

Docker Compose abstracts individual container CLI arguments into a single declarative text file configuration blueprint.

⚙️ Why use Compose?

Instead of executing 5 complex terminal statements with individual ports, networks, and environment tags, Docker Compose aggregates your environment topology into a single unified workspace blueprint template.

🛠️ Key Configuration Structural Pillars

  • services: Definition block for your application containers (e.g., frontend, database web nodes).
  • volumes: Shared block specifying named volumes to declare stateful, durable file lifecycles.
  • networks: Shared software isolation lanes connecting related application layers.
